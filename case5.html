고객 의도 기반 환불·교환 흐름 개선 및 입력 구조 재설계

자유 텍스트에 묻혀 있던 고객 의도를 선택값으로 구조화해, AS 재확인 왕복을 95% 이상 없앤 사례입니다. 기존 중복 필드를 활용하는 스펙을 직접 설계해 개발 범위를 최소화하고, SW개발팀·IT팀 협의로 빠르게 구현했습니다.

Problem

반복 확인으로 고객과 운영 모두 지치고 있었습니다

고객이 환불/교환 의도를 자유 텍스트로만 남겨 CS가 추가 확인을 해야 했습니다. 고객은 다시 1:1 문의에 답변해야 했고, 그 과정에서 처리 흐름이 지연됐습니다.

반복 확인으로 고객 피로도와 운영 리소스가 함께 증가했습니다. CRM에 답변 작성일과 사실상 중복되는 확인일시가 기록되고 있어, 실질적 의미 없이 데이터만 누적되는 구조였습니다.

What I Did

데이터 수집 단계부터 구조화했습니다

1.

데이터 품질의 출발점을 재정의했습니다
데이터 품질은 처리 단계가 아니라 수집 단계에서 결정된다고 보고, 처리 중 추가 질문을 늘리기보다 입력 구조 자체를 바꾸는 방향으로 접근했습니다.

2.

중복 필드를 제거하고 의미 있는 데이터로 교체했습니다
의미 없이 중복되던 확인일시 필드를 제거하고, 해당 위치에 고객의 요청 유형(환불 / 교환) 선택값을 배치했습니다.

3.

입력 구조를 재설계하고 개발팀과 협의해 구현했습니다
고객 문의 등록 단계에서 환불·교환 의도를 필수 선택값으로 수집하고, 그 값이 CRM에 구조화된 데이터로 저장되도록 설계했습니다. 중복되던 확인일시 필드 자리에 신규 필드를 배치하는 방식으로 개발 범위를 최소화하고, SW개발팀·IT팀과 협의해 DB 컬럼 추가 및 UI 구현을 빠르게 완료했습니다.

Evidence

3단계의 개선 과정

Evidence 1
기존 입력 구조 문제점
기존 CRM의 환불·교환 입력 화면 — 자유 텍스트 의존 구조와 중복 필드(확인일시)가 있던 상태입니다.
Evidence 2
입력 구조 재설계 화면
답변 작성일과 중복되던 확인일시 필드를 제거하고, 해당 위치에 고객 의도 데이터를 매핑했습니다.
Evidence 3
요청 유형 선택값 반영 구조
고객이 문의 등록 단계에서 환불·교환 의도를 직접 선택하도록 바꾸고, 선택된 값이 후속 처리 기준으로 바로 활용되도록 설계했습니다.

Insight

배운 점들

1
질문을 줄이는 설계가 운영 비용을 줄였습니다.
2
