시스템 제약 환경에서의 예외 처리 로직 재정의 및 Fail-soft UX 설계

배송 가능일이 다른 상품이 장바구니에 함께 담기면 주문이 차단되는 상황에서, 고객이 이유를 모른 채 이탈하는 문제를 발견하고 직접 개선을 제안했습니다. 기술적 제약으로 모든 예외 케이스를 처리하기 어려운 상황에서, 실행 가능한 최소 단위(MVP)로 방향을 전환해 고객이 주문 불가 이유를 이해하고 흐름을 이어갈 수 있도록 Fail-soft UX를 직접 기획한 사례입니다. 개발팀과의 협의를 통해 기술 제약 안에서 최선의 솔루션을 도출했습니다.
#FailSoftUX #MVP기획 #ExceptionDesign

고객은 이유를 몰랐고, 내부에도 기준이 없었습니다

배송 가능일이 다른 상품이 장바구니에 함께 담기면 주문이 차단됩니다. 그런데 고객에게는 이유가 전혀 전달되지 않아 시스템 오류처럼 인식됐습니다.
주문에 실패한 고객은 이유를 모른 채 재시도 없이 이탈했습니다. 특히 무료배송 조건(4만원)을 맞추려 여러 상품을 함께 담다가 이 상황에 걸리는 경우도 잦아 장바구니 이탈과 매출 손실로 이어졌고, 내부에서도 별도 기준 없이 "상품을 하나씩 빼보세요"라는 안내가 전부였습니다.
특히 외부 시스템(API) 응답 제약으로 인해, 기획 단계에서 의도한 수준의 세부 예외 처리 로직을 그대로 구현하기 어려운 상태였습니다.

제약 안에서 실행 가능한 최선안을 찾았습니다

1. 구현 가능성을 포함한 재설계
이상적 로직의 한계를 먼저 확인
초기에는 모든 실패 조건을 시스템적으로 구분하려 했지만, 구현 가능성과 일정 측면에서 현실적인 한계가 있음을 확인했습니다. 구현 가능성까지 포함해 예외 처리 방식 자체를 다시 설계해야 한다고 판단했습니다.
2. API 응답값 기반 MVP 설계
기술 제약을 전제로 한 실행 가능한 설계
API 응답값으로 판단 가능한 조건만을 기준으로 실패 유형을 다시 정의했습니다. 기술 제약을 전제로 한 실행 가능한 최소 단위(MVP) 해법으로 방향을 전환해 개발팀과 합의했습니다.
3. Fail-soft UX 설계
오류 상황에서도 자연스러운 흐름 유지
주문 실패 시 세부 원인을 모두 설명하지 않더라도, 왜 지금 진행할 수 없는지를 이해할 수 있도록 Fail-soft UX를 설계했습니다. 오류 상황에서 FAQ 및 추가 안내로 자연스럽게 연결되도록 흐름을 보완하고, 운영 대응 기준도 함께 정리해 운영 혼선을 줄였습니다.

어떤 성과를 얻었나요?

배송 가능일이 다른 상품이 포함된 경우, 주문 차단 사유를 별도 페이지 이동 없이 장바구니 내에서 즉시 안내하도록 설계
주문 불가 상황에서 고객이 이유를 이해하고 흐름을 이어갈 수 있도록 설계 → 즉각 이탈 방지 및 잠재 매출 손실 구조 해소
오류 인식으로 인한 고객 문의 유입을 줄이고, 기술 제약을 고려한 실행 가능한 개선안으로 개발·운영 간 합의
유사한 제약 상황이 발생하더라도 재사용 가능한 대응 구조 확보
관련 고객 문의 일 2–3건 → 0건 (구조 개선 이후)
Key Metric
일 2–3건 → 0건
오류 문의 감소 · 이탈로 인한 매출 손실 구조 해소

근거

1. 장바구니 내 Fail-soft 안내 설계
배송 가능일이 다른 상품이 포함된 경우, 별도 페이지 이동 없이 장바구니 안에서 즉시 주문 불가 사유를 확인할 수 있도록 설계했습니다.
2. 기술 제약 기반 예외 처리 재정의
기술 제약 기반 설계 자료
모든 실패 조건을 세분화하는 방식 대신, API 응답값으로 실제 판단 가능한 조건만 남겨 구현 가능한 범위 안에서 예외 처리 구조를 다시 설계했습니다.
3. 문제 발견 및 개선 제안 배경
내부 개선 제안 커뮤니케이션
배송일 충돌 케이스를 직접 파악하고, 운영팀에 개선을 제안했습니다. "모든 케이스를 안내하기 어렵다는 건 이해하지만, 고객이 최소한 왜 막히는지만 알아도 불신과 이탈이 줄 것"이라는 관점에서 UX 개선 필요성을 구체적으로 제시했고, 이 제안이 개선 착수의 계기가 됐습니다. 특히 무료배송 기준(4만원)을 맞추려다 이 상황에 걸리는 고객의 경험을 직접 짚어낸 점, 그리고 기존 안내 방식("상품 하나씩 빼보세요")이 고객 입장에서 아쉽다는 점을 근거로 제시했습니다.

배운 점

1
완벽한 로직보다 실패를 설명하는 설계가 사용자 신뢰를 만들었습니다. 모든 오류를 기술적으로 해결하기 어려울 때, 고객이 상황을 이해하고 수용할 수 있도록 하는 것이 더 효과적이었습니다.
2
기술적 제약은 포기의 이유가 아니라 우선순위 조정의 기준이 됐습니다. 제약 속에서 정말 중요한 문제가 무엇인지 다시 생각하게 했고, 그것이 더 나은 솔루션을 만들었습니다.
3
기획자의 역할은 정답을 고집하는 것이 아니라 실행 가능한 최선안을 찾는 일임을 배웠습니다. 개발팀과 함께 제약을 공유하고, 그 안에서 고객에게 최대한의 가치를 전달하는 구조를 만드는 것이 핵심이었습니다.
