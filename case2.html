"주의상품" 시스템 신설 — 배송 제약 상품의 장바구니 UX 개선

배송 가능일이 제한된 상품이 장바구니에서 "품절"로 표시되어, 하루 6~10건의 고객 문의가 반복되던 문제였습니다. 반년 전 직접 CTO에게 해결 방향을 제안했지만 미뤄졌던 이슈를, 데이터와 함께 다시 꺼내 MVP 개선을 재제안했습니다. CTO가 제시한 "주의상품" 태깅 방식을 기반으로 장바구니 내 안내 문구와 표시 방식을 직접 기획해, 제안 당일 개발·배포까지 이어진 사례입니다.
#MVP기획 #문제발견→제안→실행 #DevCollaboration #주의상품시스템

품절이 아닌데 "품절"이라고 안내되고 있었습니다

배송 제약 상품이 "품절"로 표시되는 구조적 문제
특정 요일에만 배송 가능한 상품(예: 화요일 도착 상품)을 다른 상품과 함께 담으면, 장바구니에서 "품절"로 표시됐습니다. 실제로는 재고가 있지만 배송일이 맞지 않는 것뿐인데, 시스템이 이 둘을 구분하지 못했습니다.
고객 혼란과 반복되는 문의
고객은 "품절"이라는 안내를 보고 재입고를 기다리거나, 이유를 모른 채 이탈했습니다. 대표적으로 연두부 상품 하나만으로 하루 6~10건의 문의가 발생했고, 운영팀도 "상품을 따로 주문해주세요"라고 안내하는 것이 전부였습니다.
기술적으로 쉽게 고칠 수 없는 구조
장바구니에서 실제 품절과 배송일 충돌이 동일하게 표시되고 있었고, 이 둘을 프론트엔드만으로 구분해 안내하기 어려운 구조였습니다. 이전에도 개선 논의가 있었지만, 전면 개편은 일정과 리소스 면에서 현실적이지 않아 미뤄져 온 상황이었습니다.

전면 개편 대신, 지금 실행 가능한 MVP를 제안했습니다

1. 문제 파악 및 CTO 제안
반년 전 직접 제안했던 이슈를, 데이터와 함께 다시 꺼내 재제안
반년 전 CTO에게 직접 DM으로 "충돌 시 간단한 팝업 안내만으로도 고객 혼란과 인입을 줄일 수 있다"는 해결 방향을 제안했지만, 기술적 복잡성으로 미뤄졌던 문제였습니다. 연두부 상품에서 문의가 다시 증가하는 데이터를 근거로, CTO에게 직접 개선을 다시 요청했습니다. "시스템에서 실제 품절과 배송일 불일치를 구분할 수 있다면, 안내 문구만 다르게 보여주는 MVP로 가능하지 않을까요?"라는 질문으로 실행 가능한 방향을 함께 제시했습니다.
2. MVP 범위 설정 — "주의상품" 시스템 제안
CTO의 "주의상품" 태깅 제안을 받아 구체적인 UX 방향 설계
MVP 가능성을 제안하자 CTO가 "주의상품" 태깅 방식으로 빠르게 방향을 잡았습니다. 이 방향을 받아 장바구니에서 해당 상품 옆에 바로 사유가 보이도록 표시 방식을 구체화했고, 상품별로 라벨(예: "화요일 도착")을 다르게 넣어 다른 주의상품에도 재사용할 수 있는 구조를 함께 제안했습니다.
3. 장바구니 UX 방향 설계
주의상품의 장바구니 내 표시 방식 직접 기획
주의상품이 장바구니에 담겼을 때 "⚠️ 화요일에만 도착 가능한 상품이에요. 나누어 주문이 필요할 수 있어요."라는 안내 문구를 직접 작성하고, 품절 문구 대신 표시되도록 설계했습니다. 웹에서 줄바꿈이 깨지는 이슈를 발견해 한 문장 포맷으로 수정을 제안했고, 개발팀과 논의해 장바구니 아이템의 subtext 영역을 활용하는 구현 방식까지 합의했습니다.
4. 제안 당일 개발 → 배포 완료
CTO 승인 → 개발팀 당일 구현 → 실서비스 반영
문제 제기 후 CTO가 바로 "주의상품" 방식을 제안하고 개발팀에 지시했으며, 같은 날 구현이 완료되어 실서비스에 반영됐습니다. 다음 날 동일한 배송 제약이 있는 한컵 두부 상품도 직접 확인해 추가 태깅을 요청하는 등, 유사 상품으로의 확대까지 주도했습니다.

어떤 성과를 얻었나요?

하루 6~10건 발생하던 연두부 관련 문의가 주의상품 적용 후 0건으로 해소 — 근본 원인(품절 오안내)을 구조적으로 제거
품절과 배송 제약을 구분하는 "주의상품" 시스템을 새로 신설 — 연두부 외 한컵 두부 등 유사 상품으로 확대 적용 중
전면 개편 없이 MVP 범위로 개발팀 부담을 최소화하면서 고객 경험 개선을 실현
문제 재제기부터 CTO 논의, 개발, 배포까지 당일 내 완료 — 데이터 기반 제안과 UX 기획으로 빠른 실행을 이끄어냄
Key Metric
일 6~10건 → 0건
제안 당일 배포 · 유사 상품으로 확대 적용 중

제안부터 배포까지

1. 주의상품 안내 적용 후 장바구니 화면
연두부 상품에 "⚠️ 화요일에만 도착 가능한 상품이에요"라는 주의상품 안내가 표시됩니다. 기존에는 "품절"로만 표시되던 것이 실제 사유로 바뀌어 고객이 원인을 바로 파악할 수 있습니다.
2. 문제 구조 — 왜 쉽게 고칠 수 없었나
장바구니 API가 실제 품절과 배송일 충돌을 동일한 상태값으로 반환하는 구조적 한계.

→ 프론트엔드만으로는 두 상황을 구분할 수 없어, 서버 측에서 배송 제약 상품을 별도로 태깅하는 "주의상품" 방식을 제안.
CTO가 제안한 "주의상품" 서버 태깅 방식을 기반으로, 장바구니 내 상품별 안내 문구와 재사용 가능한 라벨 구조를 직접 기획했습니다.
3. 문제 발견 및 CTO 제안 과정
연두부 상품의 반복적인 고객 문의(일 6~10건)을 근거로 CTO에게 직접 개선을 제안했습니다. 전면 개편이 아닌 "주의상품 태깅 → 장바구니 안내 문구 표시"라는 MVP 방향을 함께 제시했고, CTO 승인 후 같은 날 개발팀이 구현을 완료해 실서비스에 반영됐습니다.
4. 구현 후 품질 개선 및 유사 상품 확대
구현 직후 웹에서 문구 줄바꿈이 깨지는 이슈를 발견해 한 문장으로 다듬는 수정을 제안했습니다. 다음 날에는 유사한 구조의 상품(한컵 두부)에도 동일한 주의상품 태깅 확대를 요청했고, CTO가 즉시 반영했습니다.

배운 점

1
"왜 안 되는지"를 아는 것만으로도 고객 경험이 달라집니다. 품절이라는 잘못된 안내 하나가 재입고 대기, 이탈, CS 문의를 동시에 만들고 있었습니다. 정확한 이유를 전달하는 것 자체가 해결의 시작이었습니다.
2
전면 개편이 어려울 때, "이 정도면 가능하지 않을까?"라는 질문이 실행을 만듭니다. "고쳐주세요"가 아니라 "구분이 가능하다면 문구만 바꾸면 되지 않을까요?"라는 구체적인 질문을 던지니, CTO가 바로 방법을 찾아 당일에 움직일 수 있었습니다.
3
기획자가 문제와 해결 방향을 함께 제시하면, 개발팀이 실행에 집중할 수 있습니다. 반복 문의 데이터, 품절 오안내의 원인, 고객이 겪는 실제 경험을 구체적으로 전달했기 때문에 CTO가 "주의상품" 방식을 바로 떠올릴 수 있었고, 같은 날 실행으로 이어졌습니다.
