시스템 제약 환경에서의 예외 처리 로직 재정의 및 Fail-soft UX 설계

기술적 제약으로 이상적인 예외 처리 로직 구현이 어려운 상황에서, 실패 원인을 이해 가능하게 전달하는 UX와 실행 가능한 최소 단위의 대응 구조를 설계한 사례입니다.

Problem

어떤 문제가 있었나요?

특정 조건에서는 정상 주문이 불가능했지만, 시스템상 오류 원인이 사용자에게 충분히 전달되지 않았습니다.
고객은 이를 단순 시스템 오류로 인식하고 이탈하거나 CS 문의로 전환했고, CS 역시 명확한 기준 없이 개별 응대로 처리해야 했습니다.
특히 외부 시스템(API) 응답 제약으로 인해, 기획 단계에서 의도한 수준의 세부 예외 처리 로직을 그대로 구현하기 어려운 상태였습니다.

What I Did

어떻게 해결했나요?

1. 이상적 예외 처리의 한계를 확인
구현 가능성을 포함한 재설계
초기에는 모든 실패 조건을 시스템적으로 구분하려 했지만, 구현 가능성과 일정 측면에서 현실적인 한계가 있음을 확인했습니다. 이 과정에서 구현 가능성까지 포함해 예외 처리 방식을 다시 설계해야 한다고 판단했습니다.
2. 기술 제약을 전제로 한 MVP 해법으로 전환
API 응답값 기반 실행 가능한 설계
API 응답값으로 판단 가능한 조건만을 기준으로 실패 유형을 다시 정의했습니다. 기술 제약을 전제로 한 실행 가능한 최소 단위(MVP) 해법으로 방향을 전환했습니다.
3. Fail-soft UX를 설계
오류 상황에서 고객 이해와 자연스러운 흐름
주문 실패 시 세부 원인을 모두 설명하지 않더라도, 왜 지금 진행할 수 없는지를 이해할 수 있도록 Fail-soft UX를 설계했습니다. 오류 상황에서 FAQ 및 추가 안내로 자연스럽게 연결되도록 흐름을 보완하고, CS 대응 기준도 함께 정리해 운영 혼선을 줄였습니다.

Outcome

어떤 성과를 얻었나요?

배송 가능일이 다른 상품이 포함된 경우, 주문 차단 사유를 별도 페이지 이동 없이 장바구니 내에서 즉시 안내하도록 설계
주문 실패 상황에서 고객 혼란과 즉각적인 이탈을 줄일 수 있는 구조 마련
오류 인식으로 인한 CS 인입을 줄이고, 기술 제약을 고려한 실행 가능한 개선안으로 개발·운영 간 합의
이후 유사한 제약 상황이 발생하더라도 재사용 가능한 대응 구조 확보
관련 CS 인입 주 2–3건 → 0건 (구조 개선 이후)

Evidence

근거

1. 장바구니 내 Fail-soft 안내 설계
배송 가능일이 다른 상품이 포함된 경우, 별도 페이지 이동 없이 장바구니 안에서 즉시 주문 불가 사유를 확인할 수 있도록 설계했습니다.
2. 기술 제약 기반 예외 처리 재정의
기획 문서 캡처 준비 중
모든 실패 조건을 세분화하는 방식 대신, API 응답값으로 실제 판단 가능한 조건만 남겨 구현 가능한 범위 안에서 예외 처리 구조를 다시 설계했습니다.
기획 문서 캡처 추가 예정

Key Insights

배운 점

1
완벽한 로직보다 실패를 설명하는 설계가 사용자 신뢰를 만들었습니다. 모든 오류를 기술적으로 해결하기 어려울 때, 고객이 상황을 이해하고 수용할 수 있도록 하는 것이 더 효과적이었습니다.
2
기술적 제약은 포기의 이유가 아니라 우선순위 조정의 기준이 됐습니다. 제약 속에서 정말 중요한 문제가 무엇인지 다시 생각하게 했고, 그것이 더 강한 솔루션을 만들었습니다.
3
기획자의 역할은 정답을 고집하는 것이 아니라 실행 가능한 최선안을 찾는 일이라고 봤습니다. 개발팀과 함께 제약을 공유하고, 그 안에서 고객에게 최대한의 가치를 전달하는 구조를 만드는 것이 핵심이었습니다.
